# Changelog

All notable changes to `unearth` are documented here.
Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

### Fixed

- Improved `phpinfo_scan` on Cloudflare-protected targets by sending browser navigation headers, retrying managed-challenge responses with the Chrome-like uTLS client, and emitting a verbose diagnostic when the page remains unavailable from the scanner's network.
- Increased the phpinfo response limit from 256 KiB to 2 MiB so `SERVER_ADDR` and `LOCAL_ADDR` fields near the end of large pages are not truncated.

## [1.0.16] — 2026-07-10

### Fixed

- Fixed `spf_mx` discovery for hostname targets such as `www.gaytell.com` by querying the registrable apex first and accepting SPF mechanism qualifiers (`+`, `-`, `~`, `?`). This allows records such as `+ip4:104.223.9.26` to enter active origin validation.
- Made verbose neighbor-scan status explicit: it now reports `no_confirmed_candidates` when no validated seed exists, or emits a summary with the number of neighboring addresses scanned and confirmed.

## [1.0.15] — 2026-07-10

### Added

- Added default active `neighbor_scan` behavior inspired by `unwaf --scan-neighbors`: after an origin is confirmed, Unearth scans that IP's `/24` neighbors and validates matches with host-header scoring. Use `--scan-neighbors=false` to disable it.
- Updated `unearth version` output to show the fork branding as `unearth MTH Mod <version>`.

## [1.0.14] — 2026-07-09

### Added

- Added an `unwaf`-style end-of-run verbose summary for confirmed origins, including IP, port, method, score components, HTTP status, source techniques, and a ready-to-run `curl -H "Host: ..."` verification command.

## [1.0.13] — 2026-07-09

### Added

- Added `shodan_host`, a generic Shodan hostname-inventory discovery technique that queries `hostname:<target>` (and the bare domain for `www` targets) so Shodan-origin hits like `5.226.140.251` enter the `host_header` validation pool even when favicon and certificate pivots are empty.

## [1.0.12] — 2026-07-09

### Fixed

- Switched `favicon_hash` favicon fetching to the same browser-like uTLS client used by active origin verification so Cloudflare-protected targets can still be hashed and searched in Shodan/Censys.
- Expanded `favicon_hash` to query every declared page icon plus `/favicon.ico`, preserved Shodan numeric `ip` fields in favicon and certificate search results, and forced a refreshed Shodan cache schema so hits such as `5.226.140.251` are no longer dropped before `host_header` can validate them.
- Kept successful `favicon_hash` backend results when the other configured backend is plan-gated, so a Censys 403 no longer discards Shodan candidates.

## [1.0.11] — 2026-07-08

### Added

- Added verbose `host_header` diagnostics showing the fetched baseline and the top rejected origin-verification probes with score components and rejection reasons.

## [1.0.10] — 2026-07-08

### Changed

- Switched active origin verification baseline and candidate probes to a Chrome-like uTLS transport with HTTP/2-first fallback, matching `unwaf` more closely and avoiding Cloudflare bot/error baselines from Go's default TLS fingerprint.

## [1.0.9] — 2026-07-08

### Fixed

- Made `host_header` direct-IP verification truly direct by using a separate TLS client without target SNI, while keeping target-SNI behavior for `Host: target` probes.
- Applied the same direct-vs-host verifier split to `asn_sweep`, which reuses the active origin validator.

## [1.0.8] — 2026-07-08

### Changed

- Strengthened active origin confirmation by probing each candidate with both direct-IP access and `Host: target`, matching the verifier flow used by tools like `unwaf`.
- Added more browser-like request headers for active origin probes.

### Fixed

- Stopped discarding short origin responses before scoring when stronger identity signals such as TLS certificate overlap or title match can still confirm the IP.

## [1.0.7] — 2026-07-08

### Added

- Added an explicit JSON/JSONL `status` field on each result (`candidate` or `confirmed`) so ranked candidates are not mistaken for verified origins.
- Added verbose output that says when no confirmed origin was found and Unearth is only showing ranked candidates.

## [1.0.6] — 2026-07-08

### Fixed

- Stopped CDN detection from warning on normal missing CNAME/NS records when A/AAAA or HTTP signals already identify the CDN.
- Reduced false timeout noise from active fanout techniques by bounding `host_header`, `phpinfo_scan`, and `banner_grab` probes more tightly on large candidate sets.
- Fixed ranking aggregation so multiple hits from the same technique on one IP count as one technique contribution instead of inflating corroboration and score.

## [1.0.5] — 2026-07-08

### Added

- Added explicit `validation` metadata for actively confirmed origin candidates and table/verbose output labels that distinguish confirmed origins from passive candidates.
- Documented this repository as the MTH mod of Unearth and added candidate-vs-confirmed-origin guidance to the README.

### Changed

- Upgraded `host_header` into a scored origin validator: it probes candidate IPs across HTTP/HTTPS ports with `Host: target` and target SNI, normalizes HTML text/title, compares text similarity, TLS certificate overlap, headers, and status, then confirms matches at a 60% threshold.
- Reused the stronger Host-header validator from `asn_sweep` so prefix-sweep hits carry the same confirmation metadata.

### Fixed

- Added clearer Netlas key-rejection diagnostics and README self-test guidance for `API key not found` responses.

### Added

- `greynoise_asset` discovery technique (passive tier, weight 0.65). The seventeenth asset-discovery backend, and unlike every backend that preceded it — the eight certificate-fingerprint engines (`censys_cert`, `shodan_cert`, `fofa_cert`, `netlas_cert`, `criminalip_asset`, `binaryedge_cert`, `leakix_cert`, `onyphe_cert`), the four crawl/asset enumerators (`fullhunt_asset`, `zoomeye_asset`, `chaos_asset`, `censys_ipv6`), and the four telemetry/scan pivots (`virustotal_passivedns`, `urlscan_asset`, `otx_passivedns`, `shodan_cve`) — it is a **sensor-observed-noise** pivot. GreyNoise runs a planetary sensor mesh that records every IP it has observed generating traffic on the public internet in a rolling ~90-day window and tags each with reverse-DNS, organization, and ASN metadata. The technique queries GreyNoise's GNQL (GreyNoise Query Language) search endpoint (`GET https://api.greynoise.io/v2/experimental/gnql`, `key` header) with a single combined expression — `metadata.rdns:*<target> OR metadata.organization:"<target>"` — and emits the non-CDN, deduplicated IPs whose rDNS metadata contains the target apex or whose organization metadata matches the target name as origin candidates. Evidence is rendered preferring the most-specific signal: an rDNS-substring match becomes `rDNS <hostname>` (the strongest signal that the IP is genuinely under the target); a bare organization match becomes `org "<org-name>"`; the fallback for hits with neither metadata field is `GNQL match`. Its value is corpus diversity from a fundamentally different axis: the IP must be **currently live and noisy enough to have hit a GreyNoise sensor recently**, *and* its rDNS or organization metadata must self-advertise as part of the target's footprint — exactly the shape of a forgotten origin (`origin.`, `direct.`, `mail.`, `staging.`) whose PTR record still points under the apex and that is still serving traffic even though it was supposed to have been retired behind the CDN. Requires `GREYNOISE_API_KEY` (also accepted as `UNEARTH_GREYNOISE_API_KEY`); without it the technique skips gracefully. The GNQL search endpoint sits behind the paid Investigate / Enterprise plans, so the Community-tier deployments are degraded cleanly: `HTTP 401` is mapped to `ErrMissingAPIKey` (the canonical "your credentials are wrong" response), while `HTTP 402` / `403` / `429` and 2xx envelopes carrying `message`/`error` mentioning `quota`, `rate limit`, `upgrade`, `subscription`, `plan`, `permission`, `payment required`, `trial`, or `feature is not enabled` all degrade to `ErrTierInsufficient` — a clean skip rather than a run failure. Bare `"ok"`/`"success"`/`"no results"` messages with an empty `data` array are treated as a benign empty success so a legitimately-empty response is not misclassified as an error envelope. The success envelope is `{"complete":bool,"count":N,"data":[{"ip","metadata":{"rdns","organization","asn"}},…],"scroll":"…","message":"…"}`; pagination uses GreyNoise's `scroll` cursor and loops until `complete:true` or the scroll cursor is empty, with a hard 10-page ceiling to bound the fetch (well past any realistic origin-discovery need). Results are cached for one hour keyed on the target apex like every sibling backend; no new dependency. This change also wires the `GreyNoiseKey` field into `techniques.APIKeys`, the `GREYNOISE_API_KEY`/`UNEARTH_GREYNOISE_API_KEY` env lookups into `config.LoadAPIKeys`, the `"greynoise"` entry into `config.CredentialStatus`, the `greynoise_asset` entry into `config.knownTechniques`, the `greynoise_asset: 0.65` line into both the embedded `pkg/config/default-weights.yaml` and the parallel `configs/default-weights.yaml`, and the engine's `hasKeyFor` pre-filter (`case "greynoise_asset": return k.GreyNoiseKey != ""`). The new technique is registered under the existing registry init pattern and picked up automatically by the `unearth_discover` MCP tool via the registry. **The R35 packet originally suggested adding either a GreyNoise asset-discovery backend or, failing that, a LeakIX service-list backend; the codebase audit at R35 start confirmed `leakix_cert` was already shipped (R24, weight 0.71) and that a LeakIX service-list variant would substantially duplicate its host/IP coverage on the same provider — GreyNoise was selected as the genuine gap.**
- `shodan_cve` discovery technique (passive tier, weight 0.78). The sixteenth asset-discovery backend and the second technique to use the `SHODAN_API_KEY` credential, but intentionally orthogonal to `shodan_cert`. Where `shodan_cert` requires the candidate host to *reuse the front-door TLS certificate* (so a patched CDN-fronted edge matches but an unpatched forgotten origin behind it often does not), `shodan_cve` inverts the signal: it requires the candidate host to *expose a specific known vulnerability*. Given an operator-supplied CVE identifier (`--cve CVE-YYYY-NNNN`, mirroring the `--email-file` pattern of operator-supplied scope) and the target apex, it queries Shodan's `/shodan/host/search` endpoint with the filter `vuln:<CVE-ID> hostname:<target>` and emits the non-CDN IPs of every host Shodan has indexed under the target's hostname that its vulnerability scanner has flagged as affected by that CVE. Purpose-built for disclosure-window recon — a freshly disclosed pre-auth RCE (ScreenConnect CVE-2024-1709, ConnectWise auth bypass, or any newly-published CVE) where the operator wants to find the unpatched forgotten origin or staging host hidden behind a patched edge. The R34 packet originally suggested **Shodan CVE search** as the first choice and **LeakIX** as the fallback, pivoting whichever represented a real gap; the codebase audit at R34 start confirmed `leakix_cert` was already shipped (R24, weight 0.71), so Shodan CVE search was selected as the genuine gap. The technique uses the same `shodanSearchURL` constant as `shodan_cert` per the codebase's one-URL-constant-per-provider discipline, shares the existing `"shodan"` budget bucket so a Free-tier allowance cannot be drained by parallel runs, and degrades cleanly on plan-gated responses (`401` → `ErrMissingAPIKey`, `403` or 200-with-`upgrade`/`membership`/`access denied` → `ErrTierInsufficient`) mirroring the sibling Shodan backend. The CVE identifier is validated against the canonical `CVE-YYYY-NNNN+` regex before any HTTP call so an operator typo fails fast rather than burning a paid API call, and the technique normalizes lowercase/whitespace-padded input (`  cve-2024-1709  ` → `CVE-2024-1709`). Empty `RunOptions.CVEID` triggers a silent skip (no error, no candidates) — the operator-supplied-scope idiom established by `email_header`. Results are cached for one hour keyed on `(target, cve)` so two parallel runs against different CVE scopes on the same target do not collide. No new dependency. This change also wires `RunOptions.CVEID` and `unearth.Options.CVEID` through the engine (`Discover` and `RunTechnique` paths), adds the `case "shodan_cve": return k.ShodanAPIKey != ""` arm to the engine's `hasKeyFor` pre-filter, and exposes the `--cve` CLI flag in `cmd/unearth/internal/cli/root.go`. The new technique is registered under the existing registry init pattern and picked up automatically by the `unearth_discover` MCP tool via the registry.
- `censys_ipv6` discovery technique (passive tier, weight 0.78). The fifteenth asset-discovery backend and the second technique to use the Censys Platform credential, but intentionally orthogonal to `censys_cert`. Where `censys_cert` pivots on the target's TLS leaf-certificate SHA-256 (and so only surfaces hosts that *reuse the front-door certificate*), `censys_ipv6` pivots on the target's DNS apex via the Censys Platform `host.dns.names` CenQL field and emits only the IPv6 hits whose addresses fall outside the embedded CDN ranges. The two techniques corroborate naturally — when both surface the same address the noisy-OR ranking combines their weights — but their coverage is genuinely disjoint: a dual-stack origin whose IPv6 listener never bound the public cert (a common misconfiguration when an IPv4 frontend is migrated behind a CDN, the IPv6 listener is forgotten on a stale self-signed or internal cert, and the AAAA record is never withdrawn) is invisible to `censys_cert` but present here. The R33 packet originally suggested implementing **either** `criminalip` (Criminal IP enrichment) **or** `censys_ipv6` (Censys IPv6 asset-discovery), pivoting whichever represented a real gap; the codebase audit at R33 start confirmed `criminalip_asset` was already shipped (R20, weight 0.70), so `censys_ipv6` was selected as the genuine gap. It hits the same `POST https://api.platform.censys.io/v3/global/search/query` endpoint as `censys_cert`, sharing the existing `CENSYS_PLATFORM_PAT` credential (also accepted as `UNEARTH_CENSYS_PAT`) and the existing `"censys"` budget bucket. The CenQL query uses the wildcard form `host.dns.names="<target>" OR host.dns.names="*.<target>"` so subdomains indexed in Censys (`origin.target`, `v6.target`, forgotten dev subdomains) match alongside the apex literal; only `host.ip` is requested in the field list, and the response is filtered with `netip.Addr.Unmap` + `Is4`/`Is6` so IPv4 hits **and** IPv4-mapped IPv6 addresses (`::ffff:1.2.3.4`) are dropped — the IPv4 origin-cert-reuse case is already covered by `censys_cert` and excluding v4 here keeps the technique's signal sharp. Pagination follows the standard `next_page_token` cursor and charges the `"censys"` budget once per page so a runaway pagination loop cannot drain a Free-tier allowance. Plan/quota responses (`HTTP 401`/`403`/`429`) — all of which indicate the account tier lacks the host-search capability or the allowance is exhausted on a Free-tier account — are mapped to `ErrTierInsufficient` so the technique degrades to a clean tier-insufficient skip rather than failing the run, mirroring `censys_cert`. Results are cached for one hour like the sibling backends; the cache key commits to the target apex and to the `"platform"` API flavor so a parallel `censys_cert` hit on the same apex does not collide. No new dependency. This change also wires the `censys_ipv6` entry into `config.knownTechniques`, the `censys_ipv6: 0.78` line into both the embedded `pkg/config/default-weights.yaml` and the parallel `configs/default-weights.yaml` (the byte-identical-copy invariant the existing `TestEmbeddedAndConfigsYAMLMatch` guards), and the engine's `hasKeyFor` pre-filter (`case "censys_ipv6": return k.CensysPlatformPAT != ""`) so the technique is gated on the same PAT presence check as `censys_cert`. The `unearth_cert_fingerprint` MCP tool's wiring is intentionally unchanged — `censys_ipv6` is not a cert-fingerprint pivot — but the `unearth_discover` MCP tool picks it up automatically via the registry.
- `otx_passivedns` discovery technique (passive tier, weight 0.64). The fourteenth asset-discovery backend, and like `fullhunt_asset`, `zoomeye_asset`, `chaos_asset`, `virustotal_passivedns`, and `urlscan_asset` — and unlike the eight certificate-fingerprint backends (`censys_cert`, `shodan_cert`, `fofa_cert`, `netlas_cert`, `criminalip_asset`, `binaryedge_cert`, `leakix_cert`, `onyphe_cert`) — it is **not** a cert pivot. It queries AlienVault OTX's passive-DNS endpoint (`GET https://otx.alienvault.com/api/v1/indicators/domain/{domain}/passive_dns`, optional `X-OTX-API-KEY` header) for every historical hostname→IP observation OTX has accumulated for the target apex and emits the non-CDN, deduplicated IPs as origin candidates. **The R32 packet originally suggested assessing FOFA or Netlas as the next backend, but the codebase audit showed both were already shipped (`fofa_cert` in an earlier packet and `netlas_cert` in R19); Criminal IP, BinaryEdge, LeakIX, Onyphe, FullHunt, ZoomEye, Chaos, VirusTotal, and URLScan were all also already shipped.** OTX was selected as the next-best unshipped backend on two distinct grounds. First, **corpus diversity**: OTX's passive-DNS corpus is a different lineage than every other backend — it is the aggregate of OTX's own honeypot/sensor network, the community-submitted "pulse" indicator feeds (analyst-curated IoC bundles describing real-world campaigns), and OTX's partner DNS telemetry; the same IP can be invisible to every scanner-driven backend yet present in OTX because a defender posted an analyst pulse mentioning it, or an OTX honeypot logged a callback to it. The corpus is also long-tailed (years of retained observations) and orthogonal to VirusTotal's: VT's passive DNS comes from URL/file scanner submissions and partner DNS feeds, whereas OTX's comes from threat-intel telemetry. Second, **deployment friction**: OTX's passive-DNS endpoint is publicly accessible to anonymous callers — making `otx_passivedns` the **first and only OSINT backend in the suite that runs without credentials**. Every other keyed backend skips itself when no key is present; OTX runs anyway, lifting every key-less deployment's floor coverage. When `OTX_API_KEY` (also accepted as `ALIENVAULT_OTX_API_KEY` and `UNEARTH_OTX_API_KEY`) is supplied, the request is authenticated via the `X-OTX-API-KEY` header, lifting the anonymous per-IP rate limit and identifying the caller for OTX's plan-level allowance — but a missing key never produces `ErrMissingAPIKey`. Like `virustotal_passivedns` and `urlscan_asset`, OTX returns the IP directly under `address`, so there is no second-stage DNS fan-out (the target is never contacted; only `otx.alienvault.com` is); non-A/AAAA records (CNAME/NS/MX) in the `passive_dns` array are skipped so only routable origin IPs surface. The success envelope is `{"passive_dns":[{"address","hostname","record_type","first","last","asn"},…],"count"}`; rate-limit/quota responses (`HTTP 429`/`403`, or a 4xx carrying a `detail`/`error`/`message` envelope mentioning `throttle`/`rate limit`/`upgrade`/`forbidden`) degrade to a clean tier-insufficient skip, a supplied bad key (`HTTP 401`, or a `detail` envelope mentioning `Invalid API key`/`unauthorized`/`credentials`) degrades to a clean missing-key skip, and `HTTP 404` (OTX has no record for this indicator) degrades to a clean empty success — mirroring the existing `virustotal_passivedns` and `urlscan_asset` backends. Results are cached for one hour like the sibling backends; cache key is independent of the API key (the corpus returned is identical with or without auth — auth only changes rate-limit accounting). No new dependency. This change also wires the `OTXKey` field into `techniques.APIKeys`, the `OTX_API_KEY`/`ALIENVAULT_OTX_API_KEY`/`UNEARTH_OTX_API_KEY` env lookups into `config.LoadAPIKeys`, the `"otx"` entry into `config.CredentialStatus`, and the `otx_passivedns` entry into `config.knownTechniques`. `RequiresAPIKey()==false` means the engine's `hasKeyFor` pre-filter never gates the technique and `hasKeyFor` itself needs no new case.
- `urlscan_asset` discovery technique (passive tier, weight 0.66). The thirteenth asset-discovery backend, and like `fullhunt_asset`, `zoomeye_asset`, `chaos_asset`, and `virustotal_passivedns` — and unlike the eight certificate-fingerprint backends (`censys_cert`, `shodan_cert`, `fofa_cert`, `netlas_cert`, `criminalip_asset`, `binaryedge_cert`, `leakix_cert`, `onyphe_cert`) — it is **not** a cert pivot. It queries the URLScan.io search API (`GET https://urlscan.io/api/v1/search/?q=domain:{domain}&size=100`, `API-Key` header) for every public browser-rendered scan submission ever recorded against the target apex and emits the non-CDN, deduplicated page-serving IPs as origin candidates. Shodan Monitor was the alternative considered for this slot (13th backend assessment) and rejected: Monitor is Shodan's *alerting* product over netblocks the operator has pre-registered, not an asset-discovery surface for an unknown origin, in the same way Hunter.io is not (Hunter is an email-address axis); for unearth's job — "given a CDN-fronted domain, find the origin IP behind it" — Monitor adds nothing the existing `shodan_cert` backend doesn't already cover, because the operator has to know the netblock in advance to register an alert. URLScan.io's search endpoint, by contrast, adds the one corpus the existing twelve cert / asset / passive-DNS backends lack: the *moment-in-time browser-rendered scan record*. The eight cert engines pivot on the target's *current* leaf certificate (missing any origin that rotated certs or was decommissioned); the three host-inventory enumerators pivot on what active scan grids observed at crawl time; VirusTotal's passive-DNS corpus records resolutions observed in network telemetry. URLScan's corpus records *what a browser actually rendered* — community-submitted PhishTank entries, SOC playbook automation, manual lookups, URLScan's own crawler — so a misconfigured origin that *briefly* leaked from behind a CDN (a five-minute deploy cutover, a CDN outage, a quietly-shipped `direct.example.com` shortcut someone submitted once) is preserved here even though the cert engines, scan grids, and passive-DNS feeds never recorded it. Like `virustotal_passivedns` (and unlike `chaos_asset`, which returns subdomain *names* and resolves them itself), URLScan returns the page-serving IP directly under `page.ip`, so the technique needs no second-stage DNS resolution to convert names to addresses, keeping its DNS footprint at zero and its results deterministic for caching. Requires `URLSCAN_API_KEY` (also accepted as `UNEARTH_URLSCAN_API_KEY`); without it the technique skips gracefully. The success envelope is `{"results":[{"page":{"ip","domain","url","asnname"},"task":{"time"},"sort":[…]},…],"total","has_more"}` with `search_after` deep-cursor pagination (the cursor is derived from the last result's `sort` array — typically `[timestamp_int, scan_id_string]`); plan/quota responses (`HTTP 403`/`429`, or a 4xx/2xx body carrying a `message`/`description` envelope mentioning `rate limit`/`quota`/`upgrade`/`forbidden`) degrade to a clean tier-insufficient skip, missing/invalid key (`HTTP 401`, or a `message`/`description` envelope mentioning `Invalid API key`/`unauthorized`/`API-Key required`) to a clean missing-key skip — mirroring the existing `virustotal_passivedns` and `zoomeye_asset` backends. Pagination follows URLScan's `has_more` flag with a hard 10-page ceiling (1000 results max); results are cached for one hour like the sibling backends; no new dependency. This change also wires the engine's `hasKeyFor` pre-filter and the config `CredentialStatus`/`knownTechniques`/default-weights surfaces for `urlscan_asset`.
- `virustotal_passivedns` discovery technique (passive tier, weight 0.67). The twelfth asset-discovery backend, and like `fullhunt_asset`, `zoomeye_asset`, and `chaos_asset` — and unlike the eight certificate-fingerprint backends (`censys_cert`, `shodan_cert`, `fofa_cert`, `netlas_cert`, `criminalip_asset`, `binaryedge_cert`, `leakix_cert`, `onyphe_cert`) — it is **not** a cert pivot. It queries VirusTotal's v3 passive-DNS endpoint (`GET https://www.virustotal.com/api/v3/domains/{domain}/resolutions`, `x-apikey` header) for every historical hostname→IP observation VirusTotal has accumulated for the target apex and emits the non-CDN, deduplicated IPs as origin candidates. Hunter.io was the alternative considered for this slot (12th backend assessment) and rejected: Hunter's API is an email-address finder (people/emails under a domain), a different axis than asset discovery; VirusTotal's passive-DNS corpus, by contrast, adds the one coverage axis the existing eleven backends lack — temporal history. The seven cert engines pivot on the target's *current* leaf certificate (missing any origin that rotated certs or was decommissioned); the three asset enumerators pivot on host inventories indexed at the time of their last crawl. VirusTotal's corpus preserves observations going back years, harvested from URL scans, file submissions, and integrated DNS feeds, so a forgotten `origin.example.com` record from three years ago — one no cert engine ever touched and no asset crawler recorded — can still surface here with its last-observed date. Unlike `chaos_asset` (which returns subdomain *names* and resolves them itself), VirusTotal returns the IP directly under `attributes.ip_address`, so the technique needs no second-stage DNS resolution to convert names to addresses, keeping its DNS footprint at zero and its results deterministic for caching. Requires `VIRUSTOTAL_API_KEY` (also accepted as `VT_API_KEY` and `UNEARTH_VIRUSTOTAL_API_KEY`); without it the technique skips gracefully. The success envelope is `{"data":[{"attributes":{"ip_address","host_name","date"}},…],"meta":{"cursor"},"links":{"next"}}` with cursor-based pagination; plan/quota responses (`HTTP 403`/`429`, or a 4xx body carrying `error.code` of `QuotaExceededError`/`TooManyRequestsError`/`UserNotActiveError`, or a 2xx body with the same envelope) degrade to a clean tier-insufficient skip, missing/invalid key (`HTTP 401`, `AuthenticationRequiredError`, `WrongCredentialsError`) to a clean missing-key skip, and `HTTP 404` (no resolutions for the apex) to a clean empty success — mirroring the existing `chaos_asset` and `onyphe_cert` backends. Pagination follows VirusTotal's `meta.cursor` with a hard 25-page ceiling; results are cached for one hour like the sibling backends; no new dependency. This change also wires the engine's `hasKeyFor` pre-filter and the config `CredentialStatus`/`knownTechniques`/default-weights surfaces for `virustotal_passivedns`.
- `onyphe_cert` discovery technique (passive tier, weight 0.69). The eleventh asset-discovery backend, and the eighth independent certificate-pivot engine alongside `censys_cert`, `shodan_cert`, `fofa_cert`, `netlas_cert`, `criminalip_asset`, `binaryedge_cert`, and `leakix_cert`. Fingerprints the target's current TLS leaf certificate (SHA-256 — the same flavor Censys, FOFA, Netlas, and Criminal IP index) and queries the Onyphe datascan search API (`GET https://www.onyphe.io/api/v2/search/datascan`, `q=category:datascan tls.fingerprint.sha256:<sha256>`, `Authorization: bearer <key>` header) for every scanned service presenting that fingerprint, emitting the non-CDN, deduplicated hits as origin candidates. Onyphe is a French "cyber defense search engine" whose continuous internet-wide datascan corpus is meaningfully European-weighted and overlaps only partially with the existing seven engines (US-centric Shodan/Censys, APAC-weighted FOFA/ZoomEye, and the independents Netlas, Criminal IP, BinaryEdge, LeakIX), so a misconfigured origin that leaks its real certificate may surface in Onyphe when absent elsewhere — coverage diversity, not redundancy. Requires `ONYPHE_API_KEY` (also accepted as `UNEARTH_ONYPHE_API_KEY`); without it the technique skips gracefully. The success envelope is `{"error":0,"status":"ok","results":[{"ip","host":[…]},…],"total","page","max_page"}` (the `host` decoder tolerates both Onyphe's documented array shape and the single-string form older payloads emit, and an empty `ip` falls back to the first IP-literal entry in `host`); plan/quota responses (`HTTP 403`/`429`, or a `200` carrying a non-zero `error` field with a `text`/`message` description) degrade to a clean tier-insufficient skip and an invalid key to a clean missing-key skip, mirroring the existing `binaryedge_cert` and `leakix_cert` backends. Pagination uses the API's reported `max_page` plus a page-fill heuristic; results are cached for one hour like the sibling cert pivots; no new dependency. This change also wires the engine's `hasKeyFor` pre-filter and the config `CredentialStatus`/`knownTechniques`/default-weights surfaces for `onyphe_cert`. The two backends originally suggested for this slot — BinaryEdge and LeakIX — were already shipped (`binaryedge_cert` 0.72 and `leakix_cert` 0.71); Onyphe was selected as the next-best unshipped backend from the post-v0.1 directions on the grounds that its EU-weighted scan footprint is the most underrepresented coverage axis remaining after the seven shipped engines.

- `unearth calibrate` subcommand for data-driven weight tuning. After every discovery run, unearth now records a lightweight per-technique observation in its local cache: whether the candidate that technique surfaced was independently corroborated by another technique in the same run. `unearth calibrate` aggregates that history into per-technique precision estimates and suggested weight overrides; `--yaml` emits a ready-to-use `weights.yaml` block (low-confidence techniques commented out) for use with `--weights`, and `calibrate reset` clears the history. The suggested weight is a Beta-prior shrinkage estimate (observed corroboration rate blended toward the technique's current weight by a pseudo-count of 20), so techniques with few observations keep their defaults rather than swinging on noise; suggestions backed by fewer than 20 observations are flagged `low-confidence`. There is no external ground truth for origin correctness, so corroboration is used as the precision proxy. Recording is best-effort and never affects discovery results; `--no-cache` runs record nothing.
- `--pipeline-batch <n>` flag for bulk-target runs. When reading targets from `-l <file>` or stdin, up to `n` targets are now discovered concurrently instead of one at a time, addressing the slow sequential processing of large subdomain lists (`subfinder | unearth -l -`). A bounded worker pool runs discovery in parallel while a single writer emits outcomes in input order, so `jsonl` / `table` streaming and the accumulated `json` array stay deterministic regardless of completion order. Default `1` preserves the original strictly-sequential behavior; the existing per-target `--concurrency` (technique-level parallelism) is independent and composes with it.
- `email_header` discovery technique (passive tier, weight 0.85). Parses an operator-supplied raw email (`.eml`) with the standard-library `net/mail` package, walks the `Received:` header chain, and surfaces every public, non-CDN IP literal as an origin candidate. Email infrastructure is rarely fronted by the website's CDN, so relay IPs in the hop chain frequently expose the origin. Filters RFC1918 / unique-local, loopback, link-local, and multicast addresses plus any known CDN IP. Supplied via the new `--email-file <path>` CLI flag (and `RunOptions.EmailFile`); skips silently when no file is given. The active send-a-probe variant remains deferred (needs operator SMTP infrastructure).
- `split_dns` discovery technique (passive tier, weight 0.80). Detects partial-proxy / split-DNS misconfigurations: resolves the apex and `www` to determine whether a CDN-fronted front door exists, then flags non-CDN IPs on the apex or on commonly un-proxied siblings (`mail`, `smtp`, `ftp`, `direct`, `origin`, `backend`, `cpanel`, `webmail`) as high-confidence origin candidates. Purely DNS-based — no target contact, no API key. Yields no signal when no CDN front door is present.
- `favicon_hash` discovery technique (active tier, weight 0.75). Fetches the target's `/favicon.ico` (HTTPS with HTTP fallback), computes its MurmurHash3 using Shodan's convention — `mmh3(base64.encodebytes(favicon_bytes))` as a signed int32 — and queries Shodan (`http.favicon.hash`) and/or Censys (`services.http.response.favicons.hashes`) for hosts sharing that favicon. Either API key alone is sufficient; with neither configured the technique skips gracefully. Favicon hashes survive cert rotations and IP moves, complementing the cert-pivot techniques.
- `jarm_fingerprint` discovery technique (active tier, weight 0.70). A self-contained pure-Go port of the Salesforce JARM active TLS fingerprinting method: sends ten crafted ClientHello probes to each phase-1 candidate IP, folds the handshake responses into a 62-character fingerprint, and flags candidates whose JARM matches the target's reference JARM as likely origins. Rejects candidates matching an embedded table of known CDN-edge JARM signatures (Cloudflare, CloudFront, Fastly, Akamai) and skips CDN-range seed IPs before probing. Phase-2 candidate consumer like `host_header`; no API key and no external module dependency. Makes only TLS handshakes — no application-layer request — so it leaves no entry in the target's HTTP access logs.

### Fixed

- API-key environment variables now match the documented names. The README told users to `export CENSYS_PLATFORM_PAT`, `SHODAN_API_KEY`, `SECURITYTRAILS_API_KEY`, `VIEWDNS_API_KEY`, `FOFA_EMAIL` / `FOFA_KEY`, `NETLAS_API_KEY`, and `CRIMINALIP_API_KEY`, but the loader only read the `UNEARTH_`-prefixed aliases (`UNEARTH_SHODAN_API_KEY`, `UNEARTH_CENSYS_PAT`, …), so anyone following the docs had their keys silently ignored and every key-gated technique skipped. `LoadAPIKeys` now reads each credential from its documented, unprefixed name first and falls back to the legacy `UNEARTH_`-prefixed alias for backward compatibility; the unprefixed name wins when both are set. No behavior change for users already exporting the prefixed names.
- Replaced the `github.com/spaolacci/murmur3` dependency in `favicon_hash` with a self-contained, pointer-safe pure-Go MurmurHash3 x86_32 implementation. The vendored library's v1.1.0 release performed unsafe pointer arithmetic that tripped Go's `-race` / `checkptr` instrumentation (`fatal error: checkptr: pointer arithmetic result points to invalid allocation`), crashing the favicon test under the race detector and keeping CI red. The new implementation reads input with byte-assembly only, produces byte-for-byte identical hashes (the locked Shodan-convention value `-384845062` and the public MurmurHash3 reference vectors both pass), and removes a third-party dependency. `go test -race ./...` is now green.
- Removed an erroneous prefix (`192.230.64.0/18`) from the Imperva (Incapsula) range snapshot. That block is announced from AS33438 (StackPath/Highwinds) and is correctly claimed by the StackPath snapshot; its presence in the Imperva list made StackPath traffic in that range unreachable under the first-match-wins `ProviderForIP` lookup. A new `TestNoDuplicatePrefixAcrossProviders` registry-wide guard test now prevents any prefix from being claimed by two providers.

## [1.0.0] — 2026-05-17

### Added

**Twelve discovery techniques across three aggression tiers:**

*Passive* (never contacts the target):
- `crtsh` — Certificate Transparency log search via crt.sh with retry, backoff, and Cert Spotter fallback
- `ct_fingerprint` — Keyless cert-fingerprint pivot using kaeferjaeger SNI-IP datasets and Cert Spotter CT search
- `dns_history` — Historical DNS A/AAAA records via SecurityTrails and ViewDNS
- `spf_mx` — SPF and MX record pivot to resolve mail infrastructure IPs
- `subdomain_enum` — Wordlist-based subdomain resolution to find exposing subdomains
- `censys_cert` — Censys Platform certificate-search (API key required)

*Active* (direct connections to candidate IPs, not the target):
- `host_header` — HTTP host-header bypass validation against candidate IPs
- `banner_grab` — SSH and HTTP banner fingerprinting against candidate IPs
- `shodan_cert` — Shodan certificate-fingerprint pivot (API key required)

*Aggressive* (probes that may touch the target):
- `error_page` — Error-page leak detection on the target
- `ipv6_probe` — IPv6 exposure probe on the target

**Two-phase orchestration engine:**
- Phase 1: passive and active producers run in parallel
- Phase 2: consumer techniques (`host_header`, `banner_grab`) receive the pooled candidate IPs
- Per-technique timeout overrides via `TimeoutOverrider` interface

**Noisy-OR ranking engine:**
- Confidence scores in [0, 1] via independent technique agreement
- Corroboration count and `single_source` flag per candidate
- Configurable technique weights via YAML (`~/.config/unearth/weights.yaml`)

**CLI (`unearth`):**
- Target from positional argument or `--list` file (one per line)
- Tier selection: `--active`, `--aggressive`
- Output formats: `jsonl` (default), `json`, `table`
- Budget caps: `--max-censys`, `--max-shodan`, `--max-st`
- Cache management: `--no-cache`, `--refresh`
- `unearth version` — reports version, commit, and build date
- `unearth cache stats` / `purge` / `clear` — cache management subcommands
- Pipeline support: reads targets from stdin when piped; emits structured JSONL for tools like `jq`, `httpx`

**MCP server (`unearth-mcp`):**
- Five MCP tools over stdio transport: `unearth_discover`, `unearth_cert_fingerprint`, `unearth_dns_history`, `unearth_subdomain_enum`, `unearth_host_header_probe`
- Built with `mark3labs/mcp-go` v0.48.0 (MCP spec 2025-03-26)
- No API keys required to start; keyless techniques run automatically

**CDN detection (`pkg/cdn`):**
- IP range matching: Cloudflare, CloudFront (AWS), Fastly, Sucuri
- DNS CNAME/NS signals, HTTP response header signals
- Embedded snapshot with 24h disk-cached refresh from first-party sources
- `Refresh()` fetches fresh ranges from Cloudflare, AWS, and Fastly APIs

**Infrastructure:**
- SQLite result cache with configurable TTL per technique
- Per-endpoint rate limiting (configurable RPS and burst)
- XDG-compliant cache directories
- `CGO_ENABLED=0` builds: pure-Go binary, no system libraries required
- Cross-platform: linux/darwin × amd64/arm64 release artifacts
- GoReleaser release pipeline with version stamping via ldflags
